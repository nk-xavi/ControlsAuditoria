Evaluación de Controles y Riesgos
Categorías

LAF13

Descripción del control

Si el sistema se instala on premise el usuario no tiene acceso a la base de datos.

Posible cumplimiento

Explicar y documentar los controles de seguridad que impide esta situación.

Evidencia de cumplimiento
1. Arquitectura de software y separación de responsabilidades:
  • El sistema está diseñado como una aplicación SaaS donde los usuarios interactúan únicamente a través de la interfaz web utilizando credenciales específicas.
  • Todas las interacciones con la base de datos están gestionadas por una capa intermedia en el servidor (API o backend), lo que limita el acceso del usuario final exclusivamente a las funcionalidades permitidas por la aplicacón.
2. Aislamiento de la base de datos:
  • La base de datos está alojada en servidores segurosy gestionados por el proveedor (en este caso, nosotros), con medidas de seguridad como cifrado en reposo y en tránsito.
  • Los usuarios no tienen acceso directo a la base de datos ni a los servidores subyacentes, ya que estos están protegidos por firewalls, controles de acceso y configuraciones de red que impiden conexiones externas no autorizadas.
3. Sin acceso a archivos del software:
  • Al tratarse de un sistema SaaS, el software y los archivos relacionados están alojados en un entorno controlado por el proveedor.
  • Los usuarios no tienen acceso físico ni lógico a los servidores, ni pueden realizar modificaciones en el código ni explorar configuraciones internas.
4. Controles de autenticación y autorización:
  • El acceso al sistema requiere autenticación mediante credenciales únicas y seguras, con la posibilidad de implementar autenticación multifactor (MFA) para mayor protección.
  • Cada usuario está limitado por los permisos y roles asignados dentro del sistema, asegurando que solo puedan realizar las acciones autorizadas.
  • Los intentos de acceso no autorizados o actividades sospechosas son registrados y monitoreados para garantizar la seguridad del sistema.
